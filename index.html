01 — The Context

A personal project that became a statement.

The AI Caravan Jam was the trigger. A vibe coding session with a simple premise: show how AI can transform the design and development process. Not just accelerate it — reimagine it.

The challenge I set for myself: design a system where the same product is presented in three radically different ways, not as an aesthetic exercise, but as a real business decision system. Hyper-contextual personalization isn't a new idea. What was new was building it completely from scratch, solo, with AI as co-pilot, and documenting honestly what worked and what didn't.

The fictional product: KOVA NIMBVX PRO, a carbon-plated running shoe I designed by researching the market with Claude. The name, positioning, and technical specs all emerged from structured conversations with the model.

02 — The Buyer Personas

The product doesn't change.
The first question each buyer asks, does.

01
Specs-First Layout

The Expert

High disposable income. Knows exactly what they want. Comes to confirm technical specs, not to be convinced. Fluff is an insult. Stack height, drop, plate — in that order.

→ Add to Cart in 3 scrolls or less
02
Story-First Layout

The Explorer

Browsing and discovering. Needs to feel the product before understanding it. Brand narrative, lifestyle imagery, community reviews. Specs come later, once they're hooked.

→ Inspiration before information
03
Value-First Layout

The Price-Conscious

$229 is a lot of money. They need to justify it. Cost-per-mile, competitor comparison, durability proof. The price surfaces first — with all the context that defends it.

→ $229 / 500 km = $0.46 per km
03 — The Process
01
Definition
Brand, product & personas with Claude

Before drawing a single line, I used Claude for the strategic work. Market research on the high-performance running segment, competitive analysis (Adidas Adizero, Nike Vaporfly, On Cloudboom), and definition of the three buyer archetypes based on real e-commerce behavior.

The name KOVA NIMBVX PRO emerged from an iterative naming process — I asked Claude to explore nomenclatures that evoked speed, precision, and technology without sounding generic. The specs — carbon plate, 38mm stack, 8mm drop, 215g weight — were designed to be competitive but credible within the real market.

The key to this phase: structured prompts with business context, not generic questions. Claude isn't Google. It works best when you give it the problem frame before asking for the answer.

02
Discovery
Wireframes in Figma from the terminal via MCP

Instead of opening Figma manually and starting to draw, I connected Claude Code to the Figma MCP server directly from the terminal. The full flow on my machine:

# 1. Install the Figma MCP server
npx @anthropic-ai/mcp-figma

# 2. Start Claude Code in agent mode with Figma MCP
claude --mcp-server figma

# 3. From Claude Code, generate wireframes directly
# Claude accesses the Figma API and creates frames,
# components and layouts without leaving the conversation

With the three buyer personas defined, I asked Claude Code to generate a low-fidelity wireframe for each layout, describing the correct information hierarchy for each persona. The Figma MCP server translates those instructions into real objects inside the file. Not perfect — but incredibly useful for aligning structure before designing.

03
Design
Design system generated from the first layout

I designed the Expert PDP first — the most structured and technical version. Once that layout had a solid visual direction, I used Claude to extract a design system from it: typography, palette, spacing, reusable components, and the rules for using each one.

That system became the foundation for the other two layouts. I didn't have to start from scratch for each version — I had to decide which components to use, which to modify, and what new ones I needed to create to address each persona's needs.

Biggest lesson from this stage: design the most constrained case first. The Expert had no room for decoration — that forced system decisions that held up across the more narrative layouts.

04
Development
HTML generated with Claude Code + Node.js, from terminal

With the designs defined, I moved to development using Claude Code via terminal. The flow was direct: I described the component or section I needed, Claude generated the HTML/CSS, and I reviewed, adjusted, and iterated.

# Start Claude Code in the project directory
cd kova-nimbvx && claude

# Example prompt to generate a component:
# "Build the Expert PDP hero. Dark bg, shoe image right,
# specs table left, Add to Cart prominent above the fold.
# No carousel, no complex animations."

The first outputs were impressive at a glance and broken under the hood. Mobile completely broken. Responsiveness that worked in the exact viewport Claude assumed and nowhere else. The Pareto Principle hit hard: 80% of my time went into the last 20% of the project — pixel-level adjustments, debugging animations, making things work on iOS Safari.

There were moments where Claude Code got stuck in fix → new bug → fix → new bug cycles. I ended up debugging manually. Without solid HTML/CSS experience, the result would have been subpar. With that experience, Claude Code was a genuinely useful work partner — not a replacement, a multiplier.

Visual assets and some additional micro-interactions were generated with Gemini as a complement. Every tool has its strengths.

04 — What I Learned
≠

Impressive ≠ correct

Claude Code's first outputs looked great in screenshots. Under the hood, they were broken. Learning to read AI-generated code with a critical eye is a non-negotiable skill in an AI-assisted dev workflow.

80/20

The Pareto of vibe coding

The initial 20% was exciting and fast. The remaining 80% was manual work, surgical precision, and human experience. AI compressed total time but didn't eliminate complexity.

↑

Claude Code teaches while it executes

Multiple times, Claude implemented technical solutions I didn't know how to do myself. The result: new techniques learned in real context, not abstract tutorials. I ended the project knowing more than when I started.

⌘

The plan is the product

The most productive sessions started with clear context: what problem, for whom, what constraints, what success looks like. Vague prompts generate vague outputs. Strategic clarity amplifies AI's value.

05 — The Real Question

What's left of the designer when execution stops being the bottleneck?

Judgment remains
Knowing which problem is worth solving before generating a single line of code or design.
Business reading remains
The ability to read real context and make decisions with real consequences for real users.
Influence remains
The ability to move teams that don't report to you toward the correct output, not just the fast one.
Discernment remains
Knowing when AI output is correct and when it's dangerously plausible. Those are two very different things.

Amid all the noise about whether AI replaces designers, this project was my personal answer. Not as a manifesto, but as evidence. AI didn't replace me — it forced me to be more intentional about what I actually bring to the table.

Three perspectives. One product. Real design decisions.
